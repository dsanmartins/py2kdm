# Architecture Review GUI

## Purpose

The Architecture Review GUI provides a human-in-the-loop interface for reviewing, correcting, and approving architecture proposals generated by `py2kdm`.

The GUI does not edit the KDM file directly. Instead, it edits the architecture-enriched JSON model. The reviewed JSON can then be used as input for the KDM generator.

```text
python_model.architecture.json
        ↓
Architecture Review GUI
        ↓
python_model.reviewed_architecture.json
        ↓
kdm_pyecore_generator
        ↓
model.reviewed.kdm.xmi
```

This workflow keeps the architecture review independent from KDM serialization and preserves traceability between source code, recovered architecture elements, and the final KDM model.

---

## Starting the GUI

From the root of the `py2kdm` project, run:

```bash
python -m kdm_architecture_review.gui.main
```

The GUI opens a desktop window titled:

```text
py2kdm Architecture Review
```

---

## Opening an architecture proposal

Before using the GUI, generate an architecture proposal:

```bash
python run_pipeline.py --config configs/pymape_hierarchical.json --skip-kdm
```

This produces a file such as:

```text
outputs/pymape_hierarchical/python_model.architecture.json
```

In the GUI, click:

```text
Open proposal
```

and select the architecture JSON file.

When the file is opened, the GUI automatically:

1. loads the architecture proposal;
2. renders the architecture tree;
3. renders the architecture graph;
4. loads the component and relationship lists;
5. runs the validation/review report.

---

## Main GUI areas

The GUI is divided into three main regions.

```text
Left panel       Center panel       Right panel
Architecture     Graph view          Properties
Components                         Validation
Relationships
```

---

## Left panel

The left panel contains three tabs.

### Architecture tab

The `Architecture` tab shows the recovered architecture as a hierarchy.

For a MAPE-K system, the expected structure is:

```text
Managing Subsystem
  └── CL Manager
        └── Control Loop
              ├── Monitor
              ├── Analyzer
              ├── Planner
              ├── Executor
              └── Knowledge

Managed Subsystem
  ├── Sensor
  ├── Effector
  └── Measured Output
```

In many real or example systems, some abstractions may be missing if no explicit evidence was found. For example, the system may recover `Effector` elements but not `Sensor`, `Measured Output`, or `Reference Input`.

### Components tab

The `Components` tab lists all recovered architecture components.

Examples:

```text
distance [Monitor]
pid [Planner]
speed_executor [Executor]
speed_monitor [Monitor]
gas_brake [Executor]
Knowledge [Knowledge]
gas [Effector]
brake [Effector]
```

Each component may be marked as materialized or not materialized.

```text
✓ materialized in KDM
✗ not materialized in KDM
```

### Relationships tab

The `Relationships` tab lists architecture relationships such as:

```text
contains
mapek_flow
uses_knowledge
acts_through
subscribes_to
observes
observes_through
produces_measurement
uses_reference_input
evaluates_measured_output
```

Relationships can also be materialized or excluded from KDM generation.

---

## Graph view

The center panel displays the architecture graph.

The graph uses a domain-aware layout:

```text
Managing Subsystem
  CL Manager
    Control Loop
      MAPE-K components
      Managed-side components
Managed Subsystem
```

The `Managed Subsystem` is placed near the bottom of the view, while sensors, effectors and measured outputs are placed above it. This makes cross-boundary relationships easier to read, for example:

```text
Executor -> Effector
Monitor  -> Sensor
Monitor  -> Measured Output
```

### Graph navigation

The graph supports interactive navigation:

| Action | Effect |
|---|---|
| Mouse wheel | Zoom in / zoom out |
| Left click + drag on empty background | Pan the whole view |
| Left click + drag on a node | Move that architecture element |
| Moving a node | Redraws its relationships dynamically |

Node movement is currently visual only. Positions are not stored in the architecture JSON. This avoids polluting the architecture model with GUI layout information.

## Component properties

When a component is selected, the right panel displays its properties.

Typical fields include:

| Field | Meaning |
|---|---|
| `ID` | Stable architecture identifier. |
| `Name` | Display name of the architecture element. |
| `Role` | Architecture role, such as `Monitor`, `Planner`, `Executor`, or `Effector`. |
| `Materialize in KDM` | Whether the component will be generated in the KDM model. |
| `Implementation` | Code element used to implement the architecture abstraction. |
| `Code element kind` | Type of code element, such as function, method, class, variable, attribute, module or package. |
| `Code element ID` | Stable code element identifier. |
| `Qualified name` | Qualified name inferred from the code model. |
| `Container` | Parent module, class, package or namespace when available. |
| `Reason` | Human review explanation. |

Example:

```text
Architecture element:
  gas_brake <<Executor>>

Code element kind:
  function

Code element ID:
  function:pymape_hierarchical.control.gas_brake

Qualified name:
  pymape_hierarchical.control.gas_brake

Container:
  pymape_hierarchical.control
```

This allows the reviewer to understand whether an architecture abstraction comes from a function, method, class, variable, attribute, module or package.

---

## Relationship properties

When a relationship is selected, the right panel displays:

| Field | Meaning |
|---|---|
| `ID` | Stable relationship identifier. |
| `Type` | Relationship type, such as `contains`, `mapek_flow`, `uses_knowledge` or `acts_through`. |
| `Level` | `architectural` or `technical`. |
| `Materialize in KDM` | Whether the relationship will be generated in the KDM model. |
| `Endpoints` | Source and target elements. |

Example:

```text
gas_brake <<Executor>> --acts_through--> gas <<Effector>>
gas_brake <<Executor>> --acts_through--> brake <<Effector>>
```

In the reviewed KDM, these relationships can be represented as `structure::StructureRelationship` and, when applicable, as `core::AggregatedRelationship`.

## Validation panel

The validation panel reports the result of applying the active review rules to the currently loaded architecture JSON.

The table does not show a fixed list of all available rules. It shows only the findings that apply to the current model.

Validation runs automatically when:

1. a proposal is opened;
2. a component is edited;
3. a relationship is edited;
4. the user clicks `Validate`.

### Validation levels

| Level | Meaning |
|---|---|
| `OK` | A rule or construction was satisfied. |
| `WARNING` | The model is usable, but the user should review the finding. |
| `FORBIDDEN` | The model has a blocking consistency issue. Export is discouraged unless the user explicitly accepts it. |

The GUI uses strong colors to improve readability:

```text
OK        dark green
WARNING   orange
FORBIDDEN strong red
```

### Common warnings

The most common warnings are:

| Warning | Meaning |
|---|---|
| `ARV-W-04` | The same code element implements multiple architecture roles. |
| `ARV-W-01` | The control loop is partial, for example because `Analyzer` was not recovered. |
| `CTRL-W01` | No `Reference Input` was recovered. |
| `CTRL-W02` | No `Measured Output` was recovered. |
| `CTRL-W03` | No `Sensor` was recovered. |

These warnings are not necessarily errors. They often indicate that the architecture is partially explicit in the code.

For example, if a function named `speed` is recovered as both `Monitor` and `Executor`, the GUI reports:

```text
The same code element implements multiple architectural roles: Executor, Monitor.
```

The KDM generator will not create one element with two stereotypes. Instead, it will create two architecture components:

```text
speed_executor <<Executor>> implemented by function speed
speed_monitor  <<Monitor>>  implemented by function speed
```

## Editing workflow

The intended review workflow is:

```text
1. Open proposal JSON.
2. Inspect the architecture tree.
3. Inspect the graph.
4. Move nodes if needed for readability.
5. Select components and relationships.
6. Confirm or modify roles.
7. Mark components or relationships as materialized or not materialized.
8. Add review reasons when needed.
9. Check the validation panel.
10. Export reviewed JSON.
```

The GUI is a review layer between automatic recovery and final KDM generation.

---

## Exporting reviewed JSON

Use:

```text
Export reviewed JSON
```

This exports the full architecture JSON, including the original code model and the reviewed architecture model.

The exported file should be used as input to the KDM generator.

Recommended output path:

```text
outputs/pymape_hierarchical/python_model.reviewed_architecture.json
```

The exported JSON includes an `architecture_review` section:

```json
{
  "architecture_review": {
    "status": "reviewed",
    "source": "architecture_review_gui",
    "decision": "approved_with_changes"
  }
}
```

It may also include component or relationship changes such as:

```json
{
  "materialize": false,
  "review_status": "user_rejected",
  "review_reason": "This element should not be materialized in the architecture view."
}
```

---

## Exporting review actions

Use:

```text
Export review actions
```

This exports only the user decisions, not the full architecture model.

The result has a structure like:

```json
{
  "source": "architecture_review_gui",
  "decision": "approved_with_changes",
  "component_overrides": [],
  "relationship_overrides": []
}
```

This file is useful if review decisions should be stored separately and reapplied later.

Important: this file is not a valid input for the KDM generator because it does not contain the full model.

## Generating KDM from the reviewed JSON

After exporting the reviewed JSON, generate the reviewed KDM model:

```bash
python kdm_pyecore_generator/main.py \
  --input outputs/pymape_hierarchical/python_model.reviewed_architecture.json \
  --output outputs/pymape_hierarchical/model.reviewed.kdm.xmi
```

If the generator cannot find the KDM metamodel because of a relative path issue, pass the metamodel explicitly:

```bash
python kdm_pyecore_generator/main.py \
  --input outputs/pymape_hierarchical/python_model.reviewed_architecture.json \
  --output outputs/pymape_hierarchical/model.reviewed.kdm.xmi \
  --metamodel kdm_pyecore_generator/metamodels/kdm_1_4.ecore
```

## Verifying the reviewed KDM

After generation, inspect the reviewed KDM:

```bash
grep -n "extensionFamily\|Adaptive System Domain" \
  outputs/pymape_hierarchical/model.reviewed.kdm.xmi | head -40
```

```bash
grep -n "speed_executor\|speed_monitor\|gas_brake\|gas_effector\|brake_effector" \
  outputs/pymape_hierarchical/model.reviewed.kdm.xmi | head -100
```

```bash
grep -n "acts_through\|contains\|mapek_flow\|uses_knowledge" \
  outputs/pymape_hierarchical/model.reviewed.kdm.xmi | head -120
```

Expected checks:

```text
1. speed_executor and speed_monitor appear as different structure::Component elements.
2. They may both point to the same implementation code element.
3. gas_brake has acts_through relationships to gas and brake.
4. Managing Subsystem contains CL Manager.
5. CL Manager contains Control Loop.
6. Control Loop contains MAPE-K elements.
7. Managed Subsystem contains the recovered effectors.
```
